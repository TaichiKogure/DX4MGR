# Ver13 シナリオパラメータスタディ報告書 (2026/01/26)

## 1. 検証の目的
Ver13で導入された新ReworkモデルおよびScheduler、LatentRiskの統合環境において、Rework負荷の増大がシステム全体のパフォーマンスと完遂率に与える影響を特定する。特に「完了しない可能性」を含めた、システムの限界領域（チップポイント）を本番RUN（`run.py`）により検証する。

## 2. 検証方法
### 2.1 実行環境
- **エンジン**: DX4MGR Ver13 (Integrated Model)
- **手法**: `run.py` によるモンテカルロシミュレーション (n_trials=20)
- **期間**: 1095日間 (3年)

### 2.2 検証パラメータ (scenarios_20260126.csv)
Baseline（標準設定）に対し、以下のRework負荷を高めたシナリオを設定：

| シナリオ名 | 変更内容・意図 |
| :--- | :--- |
| **S1_Baseline** | 標準設定 (rework_load=1.3, dr2_mult=2.0) |
| **S2_HeavyRework** | 負荷増倍率を2倍以上に引き上げ (load=3.0) |
| **S3_DR2_Heavy** | DR2での顕在化リスクを極大化 (dr2_mult=6.0) |
| **S4_Rework_Spiral** | 再投入率高・減衰低 (cond_prob=0.9, decay=0.9) |
| **S5_Res_Crunch** | リソース半減下での負荷増 (eng_pool=5, load=2.0) |
| **S6_High_Load_Rework**| 高流入量 + Rework負荷増 (arrival=0.4, load=1.8) |

## 3. 検証結果と分析

### 3.1 システム崩壊の確認
全シナリオにおいて、**完了ジョブ数が極めて低く、特に過酷な条件下（S2, S4, S6）では全トライアルで完遂数が0件**となりました。

- **S1 (Baseline)**: 平均完遂数 0.05件。ほぼ全てのジョブが途中で滞留。
- **S2, S4, S6**: 完遂数 0.0件。システムが完全に機能不全に陥っている。

### 3.2 根本原因：摩擦（Friction）と増殖の「死の螺旋」
詳細ログ（`job_details_*.csv`）の分析により、以下のメカニズムでシステムが崩壊していることが判明しました。

1.  **高摩擦の常態化**: 
    Ver13の仕様により、`SMALL_EXP` 工程の並列サーバー数が 999 に固定されています。現在の摩擦計算ロジック（`linear`モデル）では、**実稼働数ではなくサーバー数に基づいて摩擦が決まるため、常に約51倍という異常な作業時間遅延**が発生しています。
2.  **Reworkによる目詰まり**: 
    DRで差し戻しが発生すると、`rework_load_factor` に従って新規ジョブ（タスク）が増殖します。これらのタスクが上記 51倍遅延の `SMALL_EXP` に再投入されることで、一つのジョブを終わらせるのに数年を要する状態になっています。
3.  **DR不達**: 
    ジョブが最初の会議（DR1）に到達する前にシミュレーション期間（3年）が終了してしまうため、後続の工程に全く仕事が流れていません。

## 4. 結論と提言

### 4.1 結論
Ver13の現行設定では、Rework負荷が少しでも高まると（あるいはBaselineの時点ですら）、摩擦モデルの副作用によってシステムが容易に「完遂不能」な状態に陥ります。これは「負担のかかり方」が線形ではなく、特定のパラメータを超えた瞬間に目詰まりが爆発する非線形な特性を持っていることを示しています。

### 4.2 提言
1.  **摩擦モデルの修正**: `WorkGate` における摩擦計算を `n_servers`（キャパシティ）ではなく、`busy_servers`（現在稼働中の人数/ジョブ数）に基づくように変更することを強く推奨します。
2.  **SMALL_EXP設定の再考**: 全員が並列で動ける設定（n=999）であっても、組織としての摩擦が常に最大値（51倍）で掛かり続けるのは非現実的です。
3.  **リソース割り当ての最適化**: Schedulerが `SMALL_EXP` の滞留を検知し、優先的にリソースを配分する等のロジック調整が必要です。
